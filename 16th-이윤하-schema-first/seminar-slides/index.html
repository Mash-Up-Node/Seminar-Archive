Node Team

왜 이번 프로젝트에서는 Zod였나

Express에서는 plain object를 바로 parse하는 흐름이 더 자연스러움
DTO class + transform보다 lean한 검증 경로를 원함
Drizzle + drizzle-zod와 함께 schema-first 흐름을 만들기 좋았고, 모두가 같은 요청/응답 형식을 보게 만들고 싶었음

같은 데이터는 어디서 봐도 같은 규칙을 가져야 했음

drizzle-zod를 쓰면 DB 정의와 validation 정의의 거리를 줄일 수 있어서, 검증·타입·문서의 기준점을 schema에 두기 좋았음

결론: 이번 프로젝트에서 내가 얻은 것

좋은 DTO보다 흔들리지 않는 데이터 규칙의 기준점이 더 중요했음
Schema는 백과 프론트가 함께 보는 요청/응답 형식의 기준점이 될 수 있었고, Orval로 협업 비용도 줄일 수 있었음
이번에는 클래스를 먼저 설계하지 않고, 데이터 규칙을 먼저 세웠음
