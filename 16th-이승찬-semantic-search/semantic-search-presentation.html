Challenges

평가와 임계값 설계

후보 검색 평가는 랭킹 중심, 서빙 단계는 threshold도 함께 설계

⚠️
완벽한 임계값은 없다
• 서비스마다 다를 뿐 아니라, 같은 도메인 내에서도 벡터 공간과 모델에 따라 임계값이 달라짐
• 임계값은 결과를 완벽하게 제어하는 수단이 아니라 필터로 생각해야 함 — 노이즈를 걷어내는 용도이지 품질을 보장하지 않음
• threshold 하나보다 top-k 랭킹 품질 검증이 더 중요함
오프라인 평가
Recall@k, MRR, nDCG로 후보 검색 품질 확인
온라인 지표
CTR, zero-result rate, 재검색률로 운영 품질 추적
튜닝 원칙
threshold보다 top-k, chunking, rerank 조합을 함께 조정
💡
왜 Recall@k / Precision@k를 봐야 하나: 검색은 top-1이 아니라 top-k를 서빙하므로, 정답이 top-k 안에 포함되는지가 핵심이다. threshold 하나로 반환 여부를 자르기 전에 top-k 안에 타겟이 들어오는지부터 확인해야 함

Indexing Strategy

KNN vs ANN 선택 전략

핵심 기준은 전체 규모보다 검색 시 실제 후보군 크기

KNN (Brute Force)
✓ 모든 벡터와 직접 비교
✓ Exact search라 recall 손실 없음
✓ 특히 pre-filter 후 후보군이 작을 때 가장 단순하고 정확
ANN (Approximate)
✓ 근사 알고리즘 (HNSW, IVF)
✓ 속도 우선, 약간의 정확도 손실
✓ 전역 검색이 크고 KNN이 실제로 느릴 때만 의미가 있음
💡
핵심 인사이트: 후보군이 작다면 KNN이 충분히 빠를 수 있다
ANN은 속도를 위해 recall을 일부 포기하는 선택이므로, 전역 검색이 크고 실제 탐색 비용이 커질 때 검토한다.
벡터 메모리 사용량은 차원에 비례해 증가하고, 실제 속도는 차원·구현체·필터 선택률에 따라 달라진다.

Optimization

필터링 전략 선택

필터 선택률에 따라 KNN과 ANN 전략이 갈린다

전략 01
Pre-filter → KNN
필터 후 후보군이 소규모일 때 적합
완전 탐색, 구현 단순, 충분히 빠름
인덱스 불필요
전략 02
ANN → Post-filter
중대규모에서 검토
top-n 안에 필터 조건의 k개가 없으면 결과 부족 가능
결과 부족 주의
전략 03
Pre-filter → ANN
필터 후 부분 집합에 ANN을 적용하면 전체 데이터 기준으로 구축된 사전 인덱스가 깨져 성능 저하
엔진에 따라 아예 미지원
엔진 지원 확인
💡
핵심은 필터 선택률 — 내 서비스·도메인에서 필터가 얼마나 효과적인지 먼저 고민해야 한다.
필터는 벡터 검색보다 더 효과적이고 정확한 후보 축소 수단이다.
선택률이 충분히 높으면 필터 후 KNN으로 끝나고, 그래도 후보군이 크게 남을 때만 ANN을 검토한다.
최근에는 동적 인덱스로 이 트레이드오프를 해결하려는 시도도 있다.

ANN → Post-filter: 결과 부족 문제
ANN으로 top-n 후보를 먼저 뽑은 뒤 필터를 적용하는 구조입니다.
문제는 필터 조건에 맞는 문서가 top-n 안에 충분히 없을 수 있다는 점입니다.

예시
top-n = 100으로 ANN 후보를 뽑았는데
그 중 필터 조건을 통과하는 문서가 3개뿐이라면
요청한 top-k = 10을 채울 수 없습니다.

완화 방법
top-n을 크게 잡으면 어느 정도 해결되지만,
필터 선택률이 낮을수록 필요한 top-n이 기하급수적으로 커집니다.
선택률이 매우 낮은 필터라면 Pre-filter → KNN이 더 단순하고 정확합니다.
Pre-filter → ANN: 사전 인덱스가 깨지는 이유
HNSW는 전체 벡터 공간을 기반으로 계층적 그래프 인덱스를 사전에 구축합니다. 각 노드 간 연결은 전역 분포와 군집 구조를 전제로 설계됩니다.

Pre-filter를 먼저 적용하면:
• 전역 벡터 집합이 부분 집합으로 축소되어 그래프 탐색 경로가 의미를 잃습니다. • 선택률이 낮을수록 인덱스와 실제 탐색 대상의 괴리가 커져 성능이 오히려 저하됩니다.

엔진에 따라 아예 미지원인 경우도 있으므로 filtered ANN 지원 방식을 반드시 확인해야 합니다.
