어댑터가 바뀌어도, 비즈니스 코드는 그대로
Hexagonal Architecture, 한국에서는 보통 '헥사고날 아키텍처'라고 부르는데요. 포트와 어댑터 패턴이라고도 합니다. 핵심 개념은 간단합니다. 전기를 떠올려보세요. 전기는 항상 220V입니다. 콘센트에 핸드폰을 꽂든, 아이패드를 꽂든, 애플워치를 꽂든 전기 자체는 바뀌지 않죠. 여기서 전기가 도메인, 즉 비즈니스 로직입니다. 콘센트가 포트 — 표준화된 인터페이스이고요. 거기에 꽂히는 기기들이 어댑터입니다. 이 다이어그램을 보시면, 가운데 도메인이 있고 양쪽에 포트가 있습니다. 왼쪽은 인바운드 — 외부에서 요청이 들어오는 쪽입니다. REST API나 스케줄러 같은 어댑터가 인바운드 포트를 통해 도메인에 접근합니다. 오른쪽은 아웃바운드 — 도메인이 데이터를 가져오거나 저장하는 쪽입니다. DB나 외부 API 같은 어댑터가 아웃바운드 포트 뒤에 숨어있죠. 핵심 약속은 이겁니다. 어댑터가 바뀌어도 비즈니스 코드는 한 줄도 안 건드린다. DB를 PostgreSQL에서 MongoDB로 바꿔도, REST API를 GraphQL로 바꿔도, 도메인 로직은 그대로입니다.

컴파일 타임에도, 런타임에도 경계가 없다
TypeScript의 타입 시스템은 두 가지 측면에서 Hexagonal의 약속을 지켜주지 않습니다. 첫째, 컴파일 타임. TypeScript는 구조적 타이핑 언어입니다. 포트가 도메인 엔티티만 반환하라고 명시해도, DB 전용 필드인 createdAt, updatedAt 같은 게 같이 따라옵니다. TypeScript는 필요한 필드가 다 있으니 OK, 추가 필드가 있어도 에러를 안 냅니다. DB 전용 데이터가 도메인까지 조용히 침투합니다. 둘째, 런타임. TypeScript의 interface는 컴파일하면 JavaScript에서 완전히 사라집니다. JavaScript에는 interface라는 개념 자체가 없거든요. NestJS의 DI 컨테이너는 런타임에 동작하는데, 토큰으로 쓸 interface가 없습니다. abstract class로 우회하지만, 구현체가 시그니처를 완벽히 안 맞춰도 통과되는 경우가 생깁니다. 결국 컴파일 타임에도, 런타임에도 경계가 보장되지 않습니다.

Hexagonal 구조를 유지하려고 모듈 경계를 엄격히 나눴는데, 현실의 비즈니스 로직은 여러 도메인을 넘나듭니다. 결국 순환 참조가 발생하고, NestJS의 forwardRef가 코드 전반에 퍼집니다. 아키텍처가 코드를 보호하는 게 아니라, 코드가 아키텍처를 유지하느라 복잡해지는 역전이 일어납니다.

구조를 이해시키는 비용 > 구조가 주는 이점
한 가지 더. 지금 실무에서 코드의 상당 부분을 AI가 작성하고 있습니다. AI가 코드를 이해하려면 컨텍스트, 즉 토큰이 필요합니다. Hexagonal은 하나의 기능이 여러 레이어에 분산되어 있어서, AI가 전체 맥락을 파악하는 데 불필요하게 많은 토큰을 소모합니다. 그리고 솔직히, 포트와 어댑터의 분리가 필요했다면 — Hexagonal이 아니어도 인터페이스 하나로 충분합니다. 아키텍처 전체를 도입할 필요가 없었던 거죠.

비즈니스 임팩트 =
고객 만족도 리소스
스타트업에서 비즈니스 임팩트는 고객 만족도 나누기 리소스라고 생각합니다. 매일 새로운 기능을 만들어서 고객에게 가치를 전달해야 하는 환경에서는, 리소스를 최대한 줄이면서 고객 만족도를 높이기 위해 태스크의 우선순위를 면밀히 고민해야 합니다. 그 관점에서 보면, 아직 오지 않은 문제를 위한 과한 설계는 리소스를 늘리는 행위입니다.
아직 오지 않은 문제를
풀려고 노력하지 않는건 어떨까요?
특히 스타트업이라면 더 그렇습니다. 매일 새로운 기능을 만들어서 고객에게 가치를 전달해야 하는 환경에서는, 그때그때의 우선순위에 집중해야 합니다. 과한 설계는 자산이 아니라 부채가 됩니다. 아직 오지 않은 문제를 풀려고 노력하지 마세요. 지금 눈앞의 문제에 집중하세요. 감사합니다.
